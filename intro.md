# Beyond Text-in-Disguise: Generating Truly Vision-Grounded Training Data for Multimodal Tool-Using Agents

## 1 Introduction

### 1.1 The Illusion of Multimodality

Multimodal large language models (MLLMs) equipped with tool-using capabilities have become a central research direction. Recent works construct training data and benchmarks where an agent receives an image, reasons over it, and calls external tools — web search, page visit, code execution — to answer complex questions. The implicit assumption is that these tasks are *inherently multimodal*: the image is essential to the reasoning process.

We challenge this assumption.

We observe that existing MLLM agent benchmarks suffer from what we term the **Vision-Grounding Gap**: the visual modality is present but not *necessary*. Images serve as decorative entry points rather than irreplaceable information sources. To the model, the image could be swapped for a text description without meaningful performance degradation. In other words, these "multimodal" agents are solving text tasks in disguise.

Why does this happen? We identify two root causes:

1. **Tool homogeneity with text agents.** The tool sets used in existing benchmarks — web_search, visit, image_search — are the same tools a text-only agent would use. There are no tools that require pixel-level visual operations (cropping, geometric measurement, image composition). The agent never needs to *manipulate* the image.

2. **Knowledge-retrievable answers.** Most questions can be answered by retrieving factual knowledge (e.g., "What year was this brand founded?"). Even when the question references visual content, the answer lies in text databases, not in the image pixels. An LLM with sufficient world knowledge can answer without seeing the image at all.

### 1.2 Diagnosing the Gap: Two Experiments

We propose two diagnostic experiments to quantify the Vision-Grounding Gap:

**Experiment A: Caption Substitution Attack.** We replace the input image with a detailed text caption generated by a VLM, then evaluate a text-only LLM with the same tool set on the same questions.

- On existing benchmarks: the caption-substituted agent achieves comparable performance to the image-based agent (drop < 5%), confirming that visual information is redundant.
- On our dataset: performance collapses, because many questions require pixel-level operations (crop a region, compute bounding box area, verify overlap after image manipulation) that cannot be performed on text descriptions.

**Experiment B: Tool Ablation — Removing code_interpreter.** We disable the code_interpreter tool (which handles all visual operations: cropping, OCR, geometric computation, image composition) and allow only retrieval tools (web_search, image_search, visit).

- On existing benchmarks: minimal performance drop, because the vast majority of questions only require retrieval tools.
- On our dataset: a large fraction of questions become unanswerable, because they fundamentally require code execution on image pixels.

These two experiments attack from complementary angles — Experiment A from the *input side* (image is replaceable) and Experiment B from the *tool side* (visual tools are removable) — and converge on the same conclusion: **existing benchmarks lack genuine vision grounding.**

### 1.3 A Taxonomy of Visual Grounding

To systematically address the gap, we propose a taxonomy that classifies tasks by **what visual information the answer depends on** — i.e., what is the visual bottleneck that makes the task impossible without the image.

We identify four types of visual grounding, ordered by increasing depth of visual engagement:

| Type | Answer Source | What Makes It Vision-Essential |
|------|--------------|-------------------------------|
| **Pixel Reading** | Content extracted from pixels | The answer is a piece of text, a count, or a color that exists only in the image pixels. A caption may approximate but cannot precisely reproduce this information (e.g., tiny text, exact object count, specific color values). |
| **Pixel Computing** | Quantity computed from pixel geometry | The answer is a measurement derived from spatial coordinates: area, distance, aspect ratio, relative position. This information does not exist in any text representation of the image. |
| **Pixel Operating** | Result observed after image transformation | The answer can only be determined after executing an image operation (crop, rotate, flip, composite, edge detection, segmentation, path planning) and observing the visual outcome. The operation itself requires pixel data. |

Orthogonally, each task may or may not require **external knowledge retrieval** (web search, page visit). This gives a binary flag `needs_knowledge` independent of the visual type.

**How this taxonomy explains existing benchmarks' weakness:** The vast majority of tasks in current MLLM agent benchmarks fall into a single cell — Pixel Reading with `needs_knowledge=true` — where the agent identifies a visual entity and then retrieves facts about it. The visual bottleneck is shallow (entity recognition), and the heavy lifting is done by text retrieval. Our dataset systematically covers all three visual types across both knowledge settings.

**Predictions for diagnostic experiments:**

| Visual Type | Caption Substitution | Tool Ablation (remove code) |
|-------------|---------------------|-----------------------------|
| Pixel Reading | Degrades (precise text/count lost) | Degrades if OCR needed |
| Pixel Computing | Collapses (no coordinates) | Collapses (no computation) |
| Pixel Operating | Collapses (no pixels to operate on) | Collapses (no code execution) |

### 1.4 Tool Design

The agent has access to 4 tools. The first three (web_search, image_search, visit) are standard retrieval tools shared with text-only agents. The fourth — `code_interpreter` — is where the visual grounding gap is closed:

| Tool | Function |
|------|----------|
| **web_search** | Search the web for factual information |
| **image_search** | Text-to-image search or reverse image search (Google Lens) |
| **visit** | Read a web page in depth |
| **code_interpreter** | Execute Python code (PIL/OpenCV/easyocr) |

`code_interpreter` provides a rich set of visual operations that text-only agents cannot use:

| Category | Operations |
|----------|-----------|
| Geometry | crop, rotate, flip, resize, zoom_in |
| Feature detection | canny_edge, hough_line, hough_circle, contour (area/perimeter) |
| Segmentation & filtering | grabcut (foreground extraction), inrange_color (HSV threshold) |
| Drawing & annotation | draw_line, draw_circle, draw_bbox |
| Color analysis | dominant color / HSV statistics, histogram equalization |
| OCR | text recognition (easyocr) |
| Measurement | area, distance, pixel statistics, coordinate localization |

This is a key differentiator: existing benchmarks' tool sets do not include any of these visual operations — their `code_interpreter` (if present) is used only for arithmetic, not for image processing.

### 1.5 Our Approach

We present an automated pipeline that generates agentic training data with coverage across all three visual grounding types. Given any collection of real-world photographs, the pipeline:

1. **Constructs a grounded knowledge graph** from each image: a VLM identifies entities and their bounding boxes; real web searches retrieve factual triples; multi-round expansion deepens the graph.

2. **Generates questions spanning all three visual types.** The knowledge graph — entities with spatial coordinates, factual triples, and inter-entity relations — is presented to an LLM, which produces questions requiring pixel reading (OCR, counting), pixel computing (area, distance, color analysis), pixel operating (edge detection, segmentation, composition, path planning), and cross-modal chains combining visual operations with knowledge retrieval.

3. **Enforces answer quality constraints.** Each question must have a unique, verifiable answer. Open-ended questions, subjective judgments, and answers requiring knowledge outside the graph are rejected.

### 1.5 Contributions

- We identify the **Vision-Grounding Gap** in existing MLLM agent benchmarks and propose two diagnostic experiments (Caption Substitution and Tool Ablation) to quantify it.
- We propose a **three-type taxonomy of visual grounding** (Pixel Reading / Computing / Operating) that provides a principled framework for evaluating whether a task truly requires vision.
- We design a fully automated pipeline with a rich visual tool set (20+ OpenCV-based operations) that generates training data covering all three visual types — the first agentic data pipeline where pixel-level visual operations are systematically required, not merely optional.
